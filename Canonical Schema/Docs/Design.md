Defining the canonical schema is the most critical technical decision of the entire project. If this data model is too rigid, it won't support complex verticals (like medical or home services). If it’s too loose, AI agents won't know how to reliably read it, and the trust layer falls apart.

This schema acts as the universal translator. Every custom connector built for a SaaS platform will map native objects into these standard entities.

Here is the blueprint for the USP Canonical Data Model (Phase 0.1).

---

### 1. Global Standards & Types

Before defining entities, the protocol must enforce strict data primitives to prevent routing and booking failures:

* **Time & Dates:** All times must be strictly ISO 8601 strings in UTC (e.g., `2026-07-08T14:00:00Z`).
* **Timezones:** IANA timezone strings (e.g., `America/Chicago`) must be explicitly defined at the Location level.
* **Currency:** ISO 4217 currency codes (e.g., `USD`).
* **Identifiers:** All USP-generated IDs should be prefixed for clarity (e.g., `mch_12345` for Merchant, `bk_9876` for Booking), while storing the original SaaS IDs separately for reconciliation.

### 2. Core Entities

#### **Merchant**

The business entity that owns the scheduling data and grants permission to USP.

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `id` | String | Yes | USP unique identifier (e.g., `mch_abc123`). |
| `external_saas_id` | String | Yes | The ID of this merchant in their native SaaS platform. |
| `saas_platform` | String | Yes | Enum (e.g., `mindbody`, `square`, `custom`). |
| `name` | String | Yes | The business name. |
| `vertical` | String | Yes | Enum (e.g., `salon`, `dental`, `hvac`). |
| `status` | String | Yes | Enum (`active`, `suspended`, `pending_consent`). |

#### **Location**

Where the service physically or virtually takes place. A Merchant can have multiple.

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `id` | String | Yes | USP location ID. |
| `merchant_id` | String | Yes | Link back to the Merchant. |
| `timezone` | String | Yes | IANA string (critical for AI time reasoning). |
| `address` | Object | No | Street, City, State, Zip, Country. |
| `is_virtual` | Boolean | Yes | Flags if this is a telehealth/Zoom meeting. |

#### **Service**

What the AI is actually trying to book.

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `id` | String | Yes | USP service ID. |
| `merchant_id` | String | Yes | Link back to the Merchant. |
| `name` | String | Yes | Consumer-facing name (e.g., "Men's Haircut"). |
| `duration_minutes` | Integer | Yes | Base duration (used by AI to block calendars). |
| `price_type` | String | Yes | Enum (`fixed`, `starting_at`, `variable`, `free`). |
| `price_amount` | Integer | No | Stored in lowest denomination (e.g., cents: 5000 = $50.00). |
| `policy_id` | String | No | Link to specific cancellation/booking rules. |

#### **Provider**

The human executing the service.

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `id` | String | Yes | USP provider ID. |
| `merchant_id` | String | Yes | Link back to the Merchant. |
| `name` | String | Yes | Name of the staff member. |
| `supported_services` | Array | Yes | List of `Service.id`s this provider can execute. |
| `status` | String | Yes | Enum (`active`, `inactive`). |

#### **AvailabilitySlot**

The most queried entity. This must be a calculated/flattened object generated by your Sync Engine, representing actual bookable time.

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `id` | String | Yes | Often a synthetic ID representing a specific time block. |
| `service_id` | String | Yes | What service this slot is for. |
| `location_id` | String | Yes | Where it happens. |
| `provider_id` | String | No | Who is providing it (null if pool-based or auto-assigned). |
| `start_time` | String | Yes | ISO 8601 UTC. |
| `end_time` | String | Yes | ISO 8601 UTC. |
| `status` | String | Yes | Enum (`available`, `held`, `booked`). |

#### **Booking**

The execution record. This tracks the state of the transaction between the AI, USP, and the native SaaS.

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `id` | String | Yes | USP generated booking ID. |
| `external_booking_id` | String | No | The ID returned by the SaaS platform (null until confirmed). |
| `idempotency_key` | String | Yes | UUID to prevent AI agents from double-booking on network retries. |
| `slot_id` | String | Yes | Link to the specific time block. |
| `customer_info` | Object | Yes | Normalized object (Name, Phone, Email). |
| `status` | String | Yes | Enum (`pending`, `held`, `confirmed`, `cancelled`, `failed`). |
| `ai_app_id` | String | Yes | Which AI agent initiated this action (for audit logs). |

#### **BookingPolicy**

The rules of engagement. AI must read this before offering a slot to a user.

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `id` | String | Yes | USP policy ID. |
| `requires_approval` | Boolean | Yes | If true, booking is `pending` until merchant manually confirms. |
| `cancellation_window_hours` | Integer | No | How many hours prior can it be cancelled without penalty. |
| `deposit_required` | Boolean | Yes | If true, blocks direct AI booking unless a payment flow is handled. |

---

### 3. The "Schema Translation" Challenge

When you build a connector, the code essentially acts as a funnel from the SaaS platform's unique language into this standard.

For example, Mindbody might return an appointment with `Staff ID = 12`, `Program ID = 5`, and a complex recurring schedule. The Connector must flatten that into an `AvailabilitySlot` with a clear `start_time`, `end_time`, `provider_id`, and `service_id` so the AI agent only has to interpret a single, clean JSON object.

This schema is bound to change over time as we stress-test this schema against platform specific edge cases.
